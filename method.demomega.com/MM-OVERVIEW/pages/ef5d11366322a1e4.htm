IDENTITY - UAF - Motivation

Description
Identifies and defines motivational elements e.g., challenges, opportunities, and concerns, that pertain to enterprise transformation efforts, and different types of equirements, e.g., operational, services, personnel, resources, or security controls.
Recommended Implementation: SysML Block Definition Diagram, SysML Package Diagram, tabular format

OMG - UAF - Asset		   Functional Asset
  Functional Building Block
 Functional Assets encompasse Architecture Blocks used to describe how an enterprise function. This includes the purposes of an enterprise (Capability) and the way the enterprise operates to fulfill these purposes (Agents and their  Behaviors).
 Functional Assets include:
1. Blocks defining results of Behaviors of the enterprise or its sub-systems, that benefit to it internal or external customers : Outcome,
2. Blocks used to describe the capabilities of the enterprise or its sub-systems: Capability,
3. Blocks used to describe information: Information Object.
4. Blocks used to describe how the enterprise operates: Operating Assets (Agent, Process, Interaction Scenario, Service Interface).
All  Functional Assets are constrained by Policys and are exposed to Risk Types.

OMG - SACM : Structured Assurance Case Metamodel		   Assurance Case
  Assured Enterprise Activity
An  Assurance Case is a claim that a particular enterprise asset or group of assets (Resources or Activites) adequately mitigates certain identified risks by means of appropriated controls.
An  Assurance Case shall provide confidence that the concerned assets will function as intended in their environment of use.
Privacy Processing Activity(ies), Data Lineages are examples of  Assurance Cases .

References:
NIST - Assurance Case
 Operating Asset   Operating Asset
  Class of Operating Block
An  Operating Asset is a Functional Asset used to describes the way Outcomes are produced and consumed: how  (Behaviors) and by whom (Agents).
 Operating Assets include  Agents and their Behaviors (Processes, Service Interfaces and Interaction Scenarios).
 Operating Assets fulfill Capability(ies) (Fulfilled Capability).
As any Functional Asset,  Operating Assets are subject to Policy Conformances. In addition, they define Rule Enforcements to indicate how Policy Conformances are met.
Any  Operating Asset can be considered as a Value Added Object which means it can be transformed and delivered as part of business transactions.
 Requirement A  Requirement specifies a capability or condition that must (or should) be satisfied. A requirement may specify a function that a system must perform or a performance condition that a system must satisfy. Requirements are used to establish a contract between the customer (or other stakeholder) and those responsible for designing and implementing the system.
References:
OMG - SysML 1.X - Requirement		   Requirement
  Requirement
A  Requirement is a generic statement used in system engineering to constrain what a Business System Asset should or should not be.
The  Requirement concept is kept in HOPEX to maintain compatibility with traditional system engineering methodologies.
The preferred approach is to follow capability-based engineering as promoted by modern enterprise & system architecture (see OMG - UAF - View - Strategic Views) and agile frameworks (see SAFe© - SAFe Requirements Model).

 Requirement (Functional) Functional requirements are represented by Capability in HOPEX.   Capability
  Capability
A  Capability  is an ability to produce an Outcome.
 Capabilitys are fulfilled by Agents performing Processes and interacting with other Agents to produce the desired Outcome.

 Requirement (Measure) Non functional requirements related to qualitative and quantitative aspects are represented by the Constraint Value concept in HOPEX.   Constraint Value
  KPI Building Block
A  Constraint Value is a possible value of a Constraining Property such as a delivery time of 30 minutes, a weight of 20 kg, a high level of confidentiality, stormy conditions.
A  Constraint Value is either a single measure (Qualifying Value (Abstract): a weight of 20 kg), an Environmental Value (stormy conditions.) or a Set of Constraint Values.
 Constraint Values are used to constrain Architecture Blocks in the context of non-functional requirement analysis and Value Proposition analysis.

 Requirement (Policy)
Non functional requirements related to constraints and guidance are represented by Policy in HOPEX.
Policies range from Regulation Articles, Business Policys to Architecture principles.
   Policy
  Policy
A  Policy is a Directive that is not directly enforceable whose purpose is to govern, guide or constrain the structure and behavior of Agents in the enterprise.
Policies provide the basis for rules and govern Behaviors carried out by Agents.
Example of  Policys are Regulation Articles, Business Policys or Architecture principles.

 Satisfy A Satisfy relationship is a dependency between a requirement and a model element that fulfills the requirement. As with other dependencies, the arrow direction points from the satisfying (client) model element to the (supplier) requirement that is satisfied.
 Satisfy (Fullfillment) Satisfy relationship that is fulfillement dependency between an operating asset and a capability.   Realized Block
 Satisfy (General) Generic satisfy relationship is a dependency between a requirement and a model element that fulfills the requirement. As with other dependencies, the arrow direction points from the satisfying (client) model element to the (supplier) requirement that is satisfied.
 Satisfy (Policy) Satisfy relationship that is realization dependency between an operating asset and a policy   Policy Conformance
  Regulatory Requirement
A  Policy Conformance is a statement that a Functional Asset is designed to comply with a Policy.
Policies can be Regulation Articles, Business Policy(ies) or Architecture principles.
