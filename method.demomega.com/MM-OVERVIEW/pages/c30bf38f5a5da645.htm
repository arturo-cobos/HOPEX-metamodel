CONCEPT DOMAIN - Hardware System ArcDevOps

Description
The  Hardware System ArcDevOps domain describes a set of concepts used in practices that combines Hardware System development (Dev) and Hardware System operations (Ops).
It aims to shorten the systems development life cycle and provide continuous delivery with high system quality. DevOps is complementary with Agile software development; several DevOps aspects came from the Agile methodology.

  Computer Device
A  Computer Device is a Computing System which provides services directly to the end-user.
 Computer Devices can host data stores and run Application code.
Examples: workstation, laptop computer, smartphone.

  Computer Server
A  Computer Server is a Computing System which provides processing, storage, and network connectivity services to other Computing Systems on a Computer Network.
As a Computational Aspect, it can host data and run Application code..

  Deployable Application Package
A  Deployable Application Package  is a split of application code according to deployment criteria at runtime. For example, it may be Front End/Back End or GUI/Business Logic etc..
Each  Deployable Application Package is associated to required Software Technology(ies) (for running) and can host code of several IT Service.
Architects can also prescribe a kind of hosting artefact (IaaS/PaaS cloud service or IT server model).

  Hardware Capability
A  Hardware Capability is a Business Operating Capability offered by Hardware Systems and aimed at delivering Physical Outcomes.
A Hardware System describes WHAT a Hardware System can provide.
 Hardware Capability(ies) are used to express the features required by people when performing their job.
For internal customers, these jobs correspond to  Operations described in Business Processes (see Instrument).
For enterprise Customers, these jobs correspond to Job-to-be-done in the context of Customer Journeys.
Examples:
- 3D Printing.
-  Automated Parcel Delivery.
- Metal Shaping.
   Hardware Functionality
  Hardware Service Interface A  Hardware Service Interface is a Business Service Interface that describes a typical course of interactions intended to produce Physical Outcomes through the involvement of Hardware Systems.   Exchange Contract
  Hardware System
A  Hardware System is a man made tangible artifact which exposes Hardware Capability(ies) and can produce Physical Outcomes.
A  Hardware System performs System Processes and participates to System Processes or to Business Processes.
A  Hardware System can embed Computational Aspects. Together with its embedded Computational Aspects, a  Hardware System can also deliver Information Outcomes.
A  Hardware System may be based on a set of Hardware Technology(ies).
Examples:
- Connected Drone with Online Payment App.
- 3D printer.
- Automated Guided Vehicles (AGVs)
- Connected fridge providing ordering Functionality and of course a freezing Hardware Capability.
- Production equipment in an assembly line (metal forging machine)
- Car

  Hardware Technology
A  Hardware Technology is an elementary Technology System  used as an enabler by Hardware Systems (as Required Hardware Technology).
 Hardware Technologys fulfill Hardware Technology Capabilitys.
Examples:
- Oxford Anor Spectrographs (NMR).
- Spectral Data Services (NMR).
- Valeo Powertrain Systems.
- Joyson Electronics Powertrain Systems.
  Hardware Technology Capability
A  Hardware Technology Capability is a Technology Capability offered by Hardware Technologys and aimed at delivering Technology Outcomes.
 Hardware Technology Capability(ies) define WHAT Hardware Technology(ies) can do, They are used to understand whether two types of Hardware Technology are fundamentally doing the same thing.
They are also used in Hardware System ArcDevOps to identify where there are redundancies in Technology Portfolios.
Exemples:
- Nuclear Magnetic Resonance (NMR) spectroscopy.
- Powertrain, Power Transmission.
- Motion Control.
  MicroService
A  MicroService is a small autonomous unit of software, emphasizing self-management and lightweightness as the means to improve software agility, scalability, and autonomy.
1)  MicroServices are automous or assembled and orchestrated as components of Applications.
2)  MicroServices can be directly deployed to Computational Aspects.  MicroServices are both a logical unit of software and a Deployable Package.
3)  MicroServices owns their own data store and dot not have any shared stores with other components.
 MicroService is Micro a enterprise asset that sits at the lower level of Software System decomposition.

Together with their Deployable Application Packages, they provide Information Outcomes.

  Physical Data Domain
A  Physical Data Domain is subset of the metadata of an enterprise’s data store.
For instance, a "Sales"  Physical Data Domain contains a "Client" Table and a "Order" Table.
Each Physical Data Entity in a  Physical Data Domain has CRUD characteristics. For instance, the "Client" Table in the "Sales"  Physical Data Domain, has all CRUD characteristics.
 Physical Data Domains define functional data boundaries used both for Data Allocation to Business Systems (see Resource Agent Store) and data governance for data stewardship (see Data Catalog).
