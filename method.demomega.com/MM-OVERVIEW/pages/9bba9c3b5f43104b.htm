  Application
An Application is a Software System that provides a set of Functionality(ies) that End Users see as a single unit.
Essentially  Applications are architectural constructions resulting from the combinaison of the following four criteria:
1) A group of Functionality that End Users see as a single unit.
2) A managed asset ( Managed Application) associated with a budget line in the context of an Application Portfolio.
3) A body of code that is seen by developers as a single unit.
4) A group of deployable software units (Deployable Application Packages) that must be installed together on one or multiple execution nodes (Computational Aspect).
 Application is a Mezzo enterprise asset  that sits between Application System and IT Service in the decomposition of Software Systems.
Example: "Payroll" is an  Application that is part an "HR system" which is an Application System.
The "Payroll"  Application includes, among other things, the "Salary and Wage Calculation" IT Service.

  Application Environment Scenario
An  Application Environment Scenario is a story involving the partners of an Application (Partner Application) interacting in order to achieve Information Outcomes in a specific Eco-System (Application Environment).
This includes:
- A course of events represented by  Application Flows depicting the story towards the delivery of expected Information Outcomes.
- Interacting Partner Applications who participate to the story in the considered Application Environment.

  Application System
An  Application System is Software System which is an assembly of multiple Applications or other  Application Systems that, together, fulfill a set of Macro Functionality(ies) delivered to Business Operations.
 Application Systems are used in the context of IT Strategic planning to define and evolve other time the hight level structure of the enterprise IT Architecture.
 Application System is a Macro enterprise asset that sit at the top of Software System decomposition.
Example: "HR System" is an  Application System whereas "Payroll" is an Application that is part of the "HR system".

  Functionality
A  Functionality is a Business Operating Capability offered by Business Systems (software or hardware) and aimed at delivering Information Outcomes.
A  Functionality describes WHAT a software or hardware system can provide.
 Functionality(ies) are used to express the features required by people when performing their job.
For internal customers, these jobs correspond to Operations described in Business Process (see Instrument)
For enterprise Customers, these jobs correspond to Job-to-be-done in the context of Customer Journeys.

  IT Service
An  IT Service is a functionnal unit of software (java class, COBOL Program, Batch) that is  a consistent, indivisible unit of processing of an Application delivering its Information Outcomes though APIs (Application Interface).
a)  IT Services are assembled and orchestrated in Applications.
b)  IT Services cannot be directly deployed to Computational Aspects: they need to be organized in Deployable Application Packages.
 IT Service is a Micro enterprise asset that sits at the lowest level of Software System decomposition.
Example: the "Salary and Wage Calculation" component is an  IT Service that is part of the "Payroll" Application.

  MicroService
A  MicroService is a small autonomous unit of software, emphasizing self-management and lightweightness as the means to improve software agility, scalability, and autonomy.
1)  MicroServices are automous or assembled and orchestrated as components of Applications.
2)  MicroServices can be directly deployed to Computational Aspects.  MicroServices are both a logical unit of software and a Deployable Package.
3)  MicroServices owns their own data store and dot not have any shared stores with other components.
 MicroService is Micro a enterprise asset that sits at the lower level of Software System decomposition.

  System Process
A  System Process is a Process that occurs inside Business Systems (software systems or hardware systems). Its performers can only be Software Systems or Hardware Systems.
During its course of action, a  System Process consumes or produces Data Objects:
1) It may read or write Data Objects in its Process Store.
2) It may receive Data Objects at its boundary: Consumed Outcome.
3) It may produce Data Objects at its boundary: Delivered Outcome.
The course of actions of a  System Process is constrained by the application of rules (System Rule Enforcement) that define what is allowed and not allowed to do.

  Physical Data Domain
A  Physical Data Domain is subset of the metadata of an enterprise’s data store.
For instance, a "Sales"  Physical Data Domain contains a "Client" Table and a "Order" Table.
Each Physical Data Entity in a  Physical Data Domain has CRUD characteristics. For instance, the "Client" Table in the "Sales"  Physical Data Domain, has all CRUD characteristics.
 Physical Data Domains define functional data boundaries used both for Data Allocation to Business Systems (see Resource Agent Store) and data governance for data stewardship (see Data Catalog).
   Physical Data Area
  Software Module
A  Software Module is a Software System that is part of an application system.
A  Software Module is either an Application, IT Service or a MicroService.
All share the ability to be composed of sub-IT Services and have dependencies to enabling Software Technology(ies).

  Software System
A  Software System is a Business System used by Business Operations, that represents all granularities of software - ranging from MicroServices to enterprise wide Application Systems - used by Business Operations.
All  Software Systems share the following characteristics:
1) They provide Functionalitys.
2) They expose APIs (Application Interfaces) through which they deliver Information Outcomes.
3) They handle datastores defined by Physical Data Domains.
4) They perform and participate to System Processes.
