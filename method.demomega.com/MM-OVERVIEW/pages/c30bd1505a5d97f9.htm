CONCEPT DOMAIN - Architecture Model Management

DescriptionPattern Domain
The  Architecture Model Management domain defines top level constructs used in enterprise architecting: Architecture Blocks, Architecture Namespaces, Architecture Category(ies), Constraint Value, Location.

ConceptDescriptionHOPEX Metaclass
  Asset Category
An  Asset Category is a general purpose Architecture Category used to classify any type of Architecture Block (Type).
The "Strategic", "Operational" and "Support" categories are examples of  Asset Category that can be applied to Capability, Process, Information Object, Policy, etc.
   Architecture Category
  Description Note A  Description Note is an annotation or remark providing a comment or an explanation on a text or a drawing.  Description Notes can be of different types: Remark, Question, Documentation, Evolution, Correction, Anomaly, Technical Specification , ...   Note
  Environmental Constraint Type
An  Environmental Constraint Type is a class of  Environmental Property.
Example: PESTEL (political, economic, social, environmental, legal),  physical conditions (weather conditions, terrain conditions,)
It is used to group Environmental Property(ies) by domain of concerns represented by Schema of Measurement.
  Environmental Property
An  Environmental Property is a characteristic of the environment in which an Enterprise or its sub-systems are required to operate.
Examples are Terrain Type (e.g. muddy, frozen ground, deep snow, etc.), light conditions (e.g. broad daylight, dusk, moonlit, etc.), weather condition, polical conditions (political stability, level of education), economical conditions (exchange rates, inflation rate).
  Environmental Value
An  Environmental Value is a specific value of an Environmental Property.
 Environmental Values are used as constraints on Architecture Blocks in the context of non-functional requirement analysis and Value Proposition analysis.
Examples:
- Muddy, frozen ground, deep snow (Terrain Type)
- Broad daylight, dusk, moonlit  (Light condition)

They allow virtual partitions of the repository. In particular, Architecture Blocks owned by different  Library(ies) can have the same name (namespacing).

  Measurable Property
A  Measurable Property is a concrete Measurable Property (Abstract) that can be applied to all Architecture Block (Type)s.
For instance, "Cost" can be applied to all Architecture Blocks, while "weight" likely applies only to Hardware Systems.

  Measure Unit
A  Measure Unit defines the units used for Measurable Property (Abstract)s.
Examples are minutes (Time Measure Type), kg (Mass Measure Type) , euros (Amount Measure Type)   etc.
  Measurement Type

  Architecture Category An  Architecture Category is a Categorization Block used to classify Architecture Blocks. Examples are Data Category, Business Line.   Category of Architecture Block
  Architecture Namespace
An  Architecture Namespace is a Model Namespace  that packages Architecture Blocks used to describe the architecture of the enterprise.
This includes  Information Asset, Processes, Software Systems, Hardware Systems, Organizations, etc.
 Architecture Namespaces are used to split HOPEX repository content into several independent modules (aka package).
They allow virtual partitions of the repository. In particular, Architecture Blocks owned by different  Architecture Namespace(ies) can have the same name (namespacing).
There are two main kinds of  Architecture Namespace: Enterprise and Library.
   Architecture Building Block Container
  Category of Category A is  Category of Category a means to classify Architecture Categorys.
  Category Partition
A  Category Partition is a split of Architecture Category(ies) in non-overlapping classifications of Architecture Block (Type)s.
For instance, the "Strategic", "Operational" and "Support" categories are exclusive categories that belongs to the "Asset Functional Classification" partition.
  Constraining Property
A  Constraining Property expresses a type of constraint that can be applied to Architecture Blocks.
These constraints can be either Measurable Property (Abstract) (Time, Mass, Cost, ..), Environmental Propertys (Terrain Type, Light conditions, Weather conditions) or a grouping of these (Set of Constraining Properties).
 Constraining Propertys are used to constrain Architecture Blocks in the context of non-functional requirement analysis and Value Proposition analysis.

  Constraint Value
A  Constraint Value is a possible value of a Constraining Property such as a delivery time of 30 minutes, a weight of 20 kg, a high level of confidentiality, stormy conditions.
A  Constraint Value is either a single measure (Qualifying Value (Abstract): a weight of 20 kg), an Environmental Value (stormy conditions.) or a Set of Constraint Values.
 Constraint Values are used to constrain Architecture Blocks in the context of non-functional requirement analysis and Value Proposition analysis.

 		  KPI Building Block
  Measurable Property (Abstract)
A  Measurable Property (Abstract) is a Constraining Property that expresses a type of measurement (Time, Mass, Cost, ..) and define the units used for measures (minutes, kg, euros, etc.).
Example:
- Time to deliver in minutes.
- Delivery cost in kilo Euros.
 Measurable Property (Abstract)s are described by Measure Unit: kg, Liter, Gallon, Hour, Minute.
HOPEX provides multiple concrete implementation of  Measurable Property (Abstract) such as Measurable Property or Flow Measure Type.
 Measurable Property (Abstract)s are used to constrain Architecture Blocks in the context of non-functional requirement analysis and Value Proposition analysis.
    References:
  •  ISO 15926 - ClassOfProperty
  •  Wikipeddia - List of quality attributes
 		  Elementary KPI Dimension
  Qualifying Value (Abstract)
A  Qualifying Value (Abstract) is a possible value of a Measurable Property (Abstract) such as a delivery time of 30 minutes, a weight of 20 kg, a high level of confidentiality.
It represents how much there is or how many there are of something that you can quantify according to its Measurable Property (Abstract).
 Qualifying Value (Abstract)s are used as constraints on Architecture Blocks in the context of non-functional requirement analysis and Value Proposition analysis.
    References:
