CONCEPT DOMAIN - Containers & Blocks

Description
The  Containers & Blocks domain defines the core elements of Building Block and Container which provide the basic for modular model management.
Blocks are reusable units. They have an independent existence. Thereby, they cannot be nested in other structures which would hide their existence (lexical scope). Because of their independent existence, they must be owned by an independent artifact which sole purpose is model management: Containers . Containers are dedicated to model management:
1) They own blocks to which they provide existence.
2) They provide namespace to blocks.
3) They may depend on other containers.

A  Block Collection is a grouping of Building Blocks.
The grouping of  Building Blocks can be achieved either by nesting, in which case the  Block Collection serves as a Container Package, or by reference.

  Building Block
A  Building Block is an autonomous, resusable Entity that contributes to the makeup of other  Building Blocks.
Building Blocks have an independent existence. Because of their independent existence, they must be owned by an independent artifact which sole purpose is model management: Containers.
Examples:
- Functionality(ies), Business Process(es), Data Entity(ies), Tags, ARE  Building Blocks.
- Operations of a process or attributes of a data entity ARE NOT building blocks. They are elements on an internal structure (see Aggregate Block).

  Container
A  Container is a collection of Building Blocks whose existence and meaning are defined by the  Container.
A  Container is both:
1. A lexical scope: the meaning of a Building Block can only be determined by the  Container where it was created.
2. A universe of discourse: the complete range of Building Blocks that are expressed, assumed, or implied in an architecture domain.
Examples:
- Container Package
- Enterprise
